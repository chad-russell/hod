# Podman Setup Guide

This guide covers using the `container-tools` profile for rootless Podman
with distrobox on Hod.

## Quick Start

```bash
# Build and activate the profile
hod profile activate profiles/container-tools.ts
source ~/.hod/profiles/container-tools/env.sh

# Verify
podman --version
podman run --rm alpine echo "Hello from podman!"
```

## How It Works

### Profile Activation

The `container-tools` profile creates a symlink farm with all binaries on PATH
and generates `containers.conf` pointing podman at the correct helper binaries
(crun, conmon, netavark, pasta). The `CONTAINERS_CONF` env var is set
automatically in `env.sh`.

### Packed Binary Handling

Some binaries (crun) use Hod's packed ELF format with AT_EXECFN-based
interpreter resolution. The profile generates wrapper scripts for these
binaries that:

1. Exec the binary from its actual store path (ensuring AT_EXECFN is correct)
2. Set `_LIBCONTAINER_CLONED_BINARY=1` to prevent crun from re-exec'ing via
   memfd (which would break packed binary resolution)

### crun Self-Clone Patch

crun normally clones itself into a memfd and re-execs as a security measure.
This is incompatible with Hod's packed binary format (AT_EXECFN would point to
`/dev/fd/X` instead of the store path). The crun recipe patches out the
self-cloning since Hod's content-addressed store provides equivalent integrity
guarantees.

## Rootless Podman Prerequisites

Podman runs rootless by default. You need:

### 1. User Namespace Configuration

```bash
# /etc/subuid — add your user
crussell:100000:65536

# /etc/subgid — add your user
crussell:100000:65536
```

### 2. newuidmap / newgidmap

These must be installed and setuid:

- NixOS: `security.wrappers.newuidmap` / `security.wrappers.newgidmap`
- Arch: `pacman -S shadow` (provides both)
- Debian/Ubuntu: `sudo apt install uidmap`

### 3. Kernel Support

Required kernel features (usually enabled by default on modern distros):

- User namespaces (`CONFIG_USER_NS`)
- Cgroups v2 (`CONFIG_CGROUPS`)
- OverlayFS (`CONFIG_OVERLAY_FS`)
- Seccomp (`CONFIG_SECCOMP`)

Verify:
```bash
# Should show user, cgroup, etc.
cat /proc/self/status | grep -i "newuid\|newgid"

# Cgroups v2
stat -f /sys/fs/cgroup
```

## Distrobox Usage

With the profile active, distrobox auto-detects podman on PATH:

```bash
# Create an Ubuntu container
distrobox-create --name dev --image ubuntu:latest

# Enter it
distrobox-enter dev

# The container integrates with your host:
# - Home directory is shared
# - Hod profile binaries are available
# - GUI apps work if the host has a display server
```

## Troubleshooting

### "could not find netavark"

The profile's `containers.conf` is loaded via `CONTAINERS_CONF` env var. Make
sure you sourced `env.sh`:

```bash
source ~/.hod/profiles/container-tools/env.sh
```

### "crun: open interp" / "hod: open interp"

This means crun's packed binary can't find its interpreter. This should not
happen with the wrapper scripts generated by the profile system. Try
re-activating:

```bash
hod profile activate profiles/container-tools.ts
```

### Permission denied on containers

Verify subuid/subgid are configured:

```bash
cat /etc/subuid
cat /etc/subgid
```

### Networking issues

pasta provides rootless networking. Verify it's on PATH:

```bash
which pasta
pasta --version
```

```bash
# Build the full stack

```

## Architecture Notes

### Why crun instead of runc?

crun is a C-based OCI runtime that's faster and lighter than runc (Go-based).
It has fewer runtime dependencies and works well in minimal environments.

### Why pasta instead of slirp4netns?

pasta is the successor to slirp4netns with better performance and features.
It's the default network mode for rootless podman.

### Why patch crun's self-cloning?

crun clones itself into a sealed memfd and re-execs. This is a security
feature for environments where the binary might be tampered with. Hod's
content-addressed store already provides this guarantee — if the store path
hashes match, the binary hasn't been modified.
